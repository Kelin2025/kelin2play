# Chat Bot

## Инструкции

### Окружение
node v15.8
npm v7.23

### Установка
1. Склонировать репозиторий
```bash
git clone git@gitlab.ecomz.net:front/chat-bot.git
```
2. Установить зависимости
```bash
npm install --legacy-peer-deps
```
3. Скопировать `.env.example` -> `.env` и прописать в нём свой `API_ROOT`

### Скрипты
- `npm start` - дев-сервер (Webpack)
- `npm run build` - продакшн сборка
- `npm run serve` - захостить билд

## Информация по проекту

### Структура проекта
Мы используем методологию **Feature Sliced**
Приложение разбито на слои:
- `shared/`
  - `lib/` - Утилити-библиотеки
  - `ui/` - UI-кит
- `entities`
  - `{entityName}` - Сущности приложения
- `features`
  - `{featureName}` - Фичи приложения
- `pages`
  - `{pageName}` - Страницы приложения

Каждая сущность/фича/страница имеет следующую структуру:
- `{featureName}`
  - `assets/` - Локальные ассеты (если нужно)
  - `lib/` - Локальные утилити-библиотеки (если нужно)
  - `model/` - Модель
  - `ui/` - Компоненты
  - `index.ts` - Индекс-файл, экспортирующий "публичное API" фичи
  - `types.ts` - Типы
### Общие принципы
- Импорты только сверху вниз. К примеру, `shared` не должны знать о `pages`
  - Также по максимуму избегаем импортов на одном уровне
  
- Бизнес-логику описываем на Effector. Избегаем `React.useState()` и прочей логики в компонентах (даже если это простой инпут)
- UI стараемся по максимуму выносить в `shared/ui`. В идеале в коде страниц/фич не должно быть стилей
- Расценивать части приложения как NPM-пакеты
  - Осознанно выносить какие-либо части приложения "наверх"
  - Прикидывать, от чего такой пакет зависел бы
  - Создавать `index.ts` с экспортом публичного API и импортировать только из него
  
- Не создавать папки `utils`, `helpers`, `hooks`
  - Вместо них - папка `lib`, в которой создаём папки с осмысленным названием и оформляем как будто делаем NPM-пакет
  
  Так проще понимать, как декомпозировать приложение, писать качественный переиспользуемый код
